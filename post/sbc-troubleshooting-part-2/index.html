SBC - Troubleshooting, Part II

July 03, 2023

The next troubleshooting session revealed yet again another bunch of schematic screw-ups and timing issues.

First of all, a closer inspection showed that all SPI devices were attached to the same SPI Slave Select signal. Another schematics copy & paste issue.

Next, the pinout for the SD card socket is totally wrong. The schematics were designed with micro SD in mind, but we changed our mind in the last minute and reverted to a regular SD socket. We only changed the socket footprint in KiCAD, and left the micro SD socket symbol as it was in the schematics. Which, as we learned now, has a different pinout.

We could at least test the VIA and verify SPI is working, but on the current board, we will never be able to use an SD card without some serious bodge-wiring, so this and the problems we found earlier make the IO section of this board completely unusable.

There is still a lot to be tested. The VDP uses two separate CS signals, one for write access, and one for read access. The separate board version uses a 74HCT139 to generate those from the GAL-generated CS_VDP and the r/W-signal from the CPU. For the SBC, we eliminated the 74HCT139 and have the CPLD generate both CS signals, CSW_VDP and CSR_VDP. In addition to the CPU bus being buffered now, this leads to a different timing. The CPLD is much faster than the GAL. In addition, the 74HCT139 also caused some delay to the CSW and CSR signals, making sure the address setup time the VDP needs is met. Now with the CPLD, and without the 74HCT139, the CS signals are pulled too early. On order to fix that, we had to come up with a slighty more sophisticated way to generate the system clock, which will be detailed in a separate post.
